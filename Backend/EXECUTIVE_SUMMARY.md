# Namma Yatri Backend: Engineering Audit Executive Summary

**Date:** March 2026
**Scope:** Backend platform (Haskell monorepo)
**Audience:** CTO / VP Engineering

---

## 1. Current State

### Architecture
Namma Yatri's backend is a Haskell monorepo comprising **48 Cabal packages** (28 application services, 15 shared libraries, 2 test suites, and utility packages) built on GHC 9.2.7. The system implements a microservices architecture organized around two principal domains: the **Rider Platform** (BAP - customer-facing: search, booking, payment) and the **Provider Platform** (BPP - driver-facing: ride management, allocation, onboarding). These communicate via the BECKN/ONDC open network protocol through Anti-Corruption Layer (ACL) modules that translate between protocol types and internal domain types.

### Scale
The codebase contains **4,495 Haskell source files** totaling **~463,000 lines of code**. Of this, approximately **122,000 lines (26%)** are auto-generated by NammaDSL from **280 YAML specifications** (75 API specs defining **298 REST endpoints**, 205 Storage specs defining **196 database tables**). The remaining **~333,000 lines** are hand-written business logic, queries, and infrastructure code. The data tier spans **PostgreSQL** (primary OLTP), **Redis** (caching, location data, KV store - referenced in 1,607 files), **Kafka** (event streaming - 191 files), and **ClickHouse** (analytics - 95 files). There are **1,412 SQL migration files** and **241 unique Haskell library dependencies**.

### Performance Architecture
The system employs a **Redis-first write pattern** via the "Drainer" architecture: high-frequency writes (driver location updates, ride state changes) land in Redis first, then asynchronously drain to PostgreSQL via dedicated drainer services (`rider-app-drainer`, `dynamic-offer-driver-drainer`). This decouples write latency from database throughput. Read-heavy queries leverage **PostgreSQL read replicas** (used in 230+ files). The driver allocation engine (`driver-offer-allocator`) runs as a separate service with its own scheduling loop, and a Redis-based job scheduler handles background task processing.

## 3. Top 5 Risks (Existential Reliability Threats)

### RISK 1: Near-Zero Test Coverage & No Tests in CI
- **Severity: CRITICAL**
- **Evidence:** Only **~54 test files** across 4 test suites (`beckn-integ-test`, `rider-app-test`, `location-updates-tests`, `yudhishthira-tests`) plus a `hunit-tests` package with 7 unit test modules — for 463K LoC (~0.01 test-to-code ratio). Only **1 property-based test** (QuickCheck). K6 load testing infrastructure exists but is minimal (single file). **Most critically: no `cabal test` execution in any CI workflow.** Of 16 GitHub Actions workflows, none run backend tests on PR or push — tests exist but are never automatically executed.
- **Impact:** Any refactoring, dependency upgrade, or code generation change has high probability of introducing undetected regressions in production. Tests that do exist provide zero value without CI execution. Zero code coverage tooling or visibility.

### RISK 2: No Distributed Tracing
- **Severity: HIGH**
- **Evidence:** Only **1 file** references tracing concepts. No OpenTelemetry, Jaeger, or Zipkin integration. Cross-service calls (rider-app -> BECKN gateway -> driver-app) are opaque.
- **Impact:** Debugging production incidents across the 10+ service boundary is manual log correlation. MTTR for cross-service failures is unnecessarily high. Cannot identify latency bottlenecks in the ride booking flow.

### RISK 3: Circuit Breakers Scoped Too Narrowly & No Bulkheads
- **Severity: HIGH**
- **Evidence:** A sophisticated circuit breaker exists (`PTCircuitBreaker.hs`) with Redis-backed failure tracking, canary probe slots, and history logging — but it is **scoped exclusively to Public Transport APIs** (Metro, Bus, Subway). Core ride-hailing integrations (Juspay payments, OSRM routing, Idfy verification, Google Maps, SMS/FCM providers) have **zero circuit breaker protection**. No bulkhead or backpressure patterns anywhere.
- **Impact:** A single degraded external service (e.g., OSRM timeout spike) can cascade into full system unavailability. The PT circuit breaker proves the team can build this — it just hasn't been generalized.

### RISK 4: Drainer Resilience Gaps
- **Severity: HIGH**
- **Evidence:** The Redis -> PostgreSQL drainer is a critical path component with a Redis stream-based error queue (`addValueToErrorQueue`) and a force-drain mode for recovery — showing thoughtful design. However: no alerting on drainer lag, no backpressure mechanism when Redis accumulates, and no monitoring of error queue depth. Redis is not durable by default.
- **Impact:** Drainer failure or sustained lag risks data loss or split-brain state between Redis and PostgreSQL. Error queue exists but without alerting, failures can go unnoticed.

### RISK 5: Missing Payment/Booking Idempotency
- **Severity: CRITICAL**
- **Evidence:** Only **5 files** reference idempotency concepts, all in the payment library. The core booking confirmation flow (ride confirm, payment capture) lacks idempotency keys. Retried requests can create duplicate bookings or double-charge riders.
- **Impact:** Network retries, client timeouts, or load balancer replays on the confirm/payment endpoints can cause financial discrepancies and broken ride state. This is a data integrity risk with direct revenue impact.

> **Note on Feature Flags:** The codebase includes **ConfigPilot** (`lib/shared-services/src/ConfigPilotFrontend/`) with 388 usage references, supporting dynamic configuration, A/B testing, and feature toggles. This is more capable than initially assessed — the gap is in deployment-level progressive rollout (canary, traffic shifting), not feature toggling itself.

## 6. Cost Optimization Opportunities

| Area | Current State | Optimization | Est. Monthly Savings |
|------|--------------|-------------|---------------------|
| **Redis Memory** | Redis used as primary write buffer (KV store for all mutations) + cache + location data. 1,607 files reference Redis. | Audit key TTLs, remove stale keys, separate location cache from write buffer, use Redis Cluster sharding. | 15-25% of Redis infra cost |
| **Read Replica Queries** | 230 files use `runInReplica`, but many queries default to primary. | Route all read-only dashboard, analytics, and list queries to replicas. Audit `Storage/Queries/` for primary-only reads that could use replicas. | 20-30% of RDS primary cost |
| **ClickHouse Query Optimization** | 95 files reference ClickHouse with unclear query patterns. | Audit materialized views, partition strategies, and query patterns. Consolidate redundant event tracking. | 10-20% of ClickHouse cost |
| **Code Generation Build Time** | Full `cabal build all` rebuilds auto-generated code. | Incremental generation (already supported via `, run-generator`), pre-compiled read-only modules, split packages for independent compilation. | Dev productivity: 15-30 min/day per developer |
| **Kafka Topic Consolidation** | Multiple fine-grained topics across services. | Audit topic utilization, merge low-volume topics, tune retention policies. | 10-15% of Kafka cost |

**Estimated Total:** 15-25% reduction in infrastructure spend (exact savings depend on current provisioning).

## 7. Rust Migration Business Case

### Assessment: **Not Recommended at This Time**

| Factor | Analysis |
|--------|----------|
| **Current Performance** | The Haskell stack with the Drainer pattern, read replicas, and Redis caching provides adequate throughput. No evidence of CPU/memory bottlenecks being the limiting factor. |
| **Migration Cost** | 333K lines of hand-written Haskell + 122K auto-generated. At $150/hr senior Rust developer rates, estimated 18-24 months and $3-5M for a partial rewrite of core services only. |
| **NammaDSL Lock-in** | The code generator produces Haskell-specific output (Beam types, Servant APIs). Retargeting to Rust would require rewriting the generator - effectively a second migration. |
| **Talent Market** | Haskell developers are scarce (~5K globally), but the team has existing expertise. Rust developers are more available (~30K) but have no domain context. Net hiring advantage is marginal. |
| **Type Safety** | Haskell's type system is equally or more expressive than Rust's for business logic (higher-kinded types, type classes, phantom types). Migration would not improve safety. |
| **Where Rust Makes Sense** | **Targeted extraction** of latency-sensitive, CPU-bound components: (1) driver allocation algorithm, (2) location processing/snap-to-road, (3) real-time fare calculation engine. These are <5% of codebase but handle >50% of compute load. |

### Recommended Approach
**Phase 1 (Months 1-3):** Extract driver location processing into a Rust microservice behind gRPC. Benchmark against current OSRM integration.
**Phase 2 (Months 4-6):** If Phase 1 shows >3x throughput improvement, extract allocation engine.
**Phase 3 (Months 7-12):** Evaluate fare calculation engine extraction based on Phase 1-2 learnings.

**ROI for targeted extraction:** 30-50% compute cost reduction for extracted services, at 10% of full migration cost.

## Key Takeaway

Namma Yatri's backend is built on **exceptionally strong foundations** - Haskell's type safety, the NammaDSL code generation system, the Redis-first write architecture, and pockets of genuine resilience engineering (PT circuit breaker, ConfigPilot, drainer error queues) are competitive advantages that most mobility platforms lack. However, the platform has **grown faster than its reliability engineering practices**. The near-zero test coverage, absent distributed tracing, missing payment idempotency, and failure isolation patterns that exist but haven't been generalized represent significant risks for a platform handling real-time ride matching and payments at scale.

The recommended roadmap prioritizes **risk reduction first** (Q1-Q2) before **acceleration** (Q3-Q4). The total investment of ~$1M over 12 months is modest relative to the platform's operational scale and would bring reliability from **3.8/10 to 8/10** - the difference between "it works until it doesn't" and "we can sleep through the night."

**The single highest-ROI action is adding idempotency to payment/booking endpoints** — this is a data integrity risk with direct revenue impact that can be fixed in under a week. The second priority is Q1 observability: you cannot improve what you cannot measure, and today the team is essentially flying blind across service boundaries.
