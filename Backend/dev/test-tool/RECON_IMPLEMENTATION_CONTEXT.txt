Reconciliation Testing Implementation Context

Purpose
- This file is the build context for implementing reconciliation testing in the local test dashboard.
- It consolidates the final agreed direction, actual table shapes observed from sample data, dependency resolution, and the implementation sequence.

Core Decisions
- Reuse the existing reconciliation scheduler trigger API.
- Do not create a second reconciliation trigger mechanism inside the test tool.
- Use the context API for local DB-backed helper endpoints:
  - readiness checks
  - finance verification
  - reconciliation result reads
  - job status polling
  - local settlement seeding
- Build one generic reconciliation flow in the dashboard, parameterized by reconciliation type.
- Keep ride/subscription-based reconciliation product-native.
- Replace SFTP/settlement-ingestion dependency with local seed data for PG-based reconciliation.

Provider Dashboard Trigger Config
- provider-dashboard runs locally on:
  - http://localhost:8018
- dashboard scheduler trigger path:
  - /bpp/driver-offer/{merchantShortId}/{city}/merchant/scheduler/trigger
- scheduler trigger payload:
  - scheduledAt
  - jobName = ReconciliationTrigger
  - jobData = JSON string containing merchantId, merchantOperatingCityId, startTime, endTime, reconciliationType
- dashboard auth header:
  - token
- the token for dashboard trigger mode must be:
  - JUSPAY_ADMIN role token
- it must not use:
  - driver registration token
  - driverOfferBpp service token
- context-api env vars:
  - RECON_TRIGGER_API_BASE_URL=http://localhost:8018
  - RECON_TRIGGER_API_TOKEN=<optional override; otherwise resolved from atlas_bpp_dashboard.registration_token>
- optional override:
  - RECON_TRIGGER_API_URL can be set to a fully qualified trigger URL if route resolution should be bypassed

Prerequisite Logic Per Reconciliation Type

DSR_VS_LEDGER
- ledger writes enabled
- completed rides exist in date window
- cancelled rides exist in date window
- booking/ledger rows present

DSSR_VS_SUBSCRIPTION
- active subscription purchases exist
- subscription credit ledger rows exist
- payment for subscription has been charged

DSR_VS_SUBSCRIPTION
- active subscription purchases exist
- completed rides linked to subscription usage exist
- subscription debit ledger rows exist

PG_PAYMENT_SETTLEMENT_VS_SUBSCRIPTION
- local mode:
  - settlement rows exist in pg_payment_settlement_report
  - rows reference real local subscription_purchase ids
- live mode:
  - would depend on settlement ingestion
- for local dashboard work, default to local seeded mode

PG_PAYOUT_SETTLEMENT_VS_PAYOUT_REQUEST
- local mode:
  - payout settlement rows exist
  - rows reference real payout_request ids
- live mode:
  - would depend on settlement ingestion
- implementation should be blocked on one real payout sample row before finalizing canonical fixture

Seed Strategy

PG payment settlement
- implement now
- canonical local fixture should follow the real sample shape above
- seed payload should accept:
  - subscriptionPurchaseId
  - optional settlementId
  - optional txnDate
  - optional rrn
  - optional paymentGateway
  - optional paymentMethod

- copy merchant_id and merchant_operating_city_id from subscription
- set reference_id = subscription_purchase.id
- set reference_type = SUBSCRIPTION_PURCHASE
- set txn_amount = subscription.plan_fee
- set txn_status = SUCCESS
- set txn_type = ORDER
- set txn_date on same UTC day as subscription.purchase_timestamp
- set recon_status = PENDING

Expected Risks

DSR_VS_LEDGER
- may produce HIGHER_IN_TARGET on floating-point equality edge cases
- cancellation cases may produce MISSING_IN_TARGET depending on ledger path completeness

DSR_VS_SUBSCRIPTION
- actual entries may still show HIGHER_IN_TARGET on near-zero variance due to rounding

PG_PAYMENT_SETTLEMENT_VS_SUBSCRIPTION
- join requires reference_id to match subscription_purchase.id exactly
- status/date constraints matter:
  - settlement txn_status must be SUCCESS
  - subscription status must be ACTIVE, EXPIRED, or EXHAUSTED
  - reconciliation code checks txnDate and purchaseTimestamp on the same UTC day

PG_PAYOUT_SETTLEMENT_VS_PAYOUT_REQUEST
- join requires payout_request_id to match payout_request.id exactly
- payout_request.status must be CREDITED
- reconciliation code checks txnDate and payout_request.createdAt on the same UTC day

RECONCILIATION PLAN — Local finance harness, ledger-first flows, dev tooling
================================================================================

Why this section
- Finance and reconciliation work should not re-invent small world-setup steps for every feature or bugfix.
- Local dev already has nix + dhall-backed processes (Postgres, Redis, Kafka, mock services, driver-offer-allocator, producer-exe, provider-dashboard, dynamic-offer-driver-app, etc.); the test-tool should lean on those same configs and URLs.
- Goal: one place to simulate base flows, call APIs as JUSPAY_ADMIN, and run read-only DB checks.

Generic principle
- New finance features should add scenario steps or assertions on top of a shared harness, not one-off scripts each time.
- Smaller flows (rides, subscriptions, cancellations, external payments) are reusable presets that can be run in one go before testing the actual change.

Ledger-first build order (prerequisites + simulation)
0. Prerequisites
   - Enable ledger (and related transporter/city flags) for the merchant operating city selected in the test dashboard.
1. Take rides and check ledgers — validate finance_ledger_entry (and links) for completed ride path.
2. Purchase subscription and check ledgers — subscription_purchase + subscription credit ledger entries; foundation for DSSR_VS_SUBSCRIPTION without SQL seeding when the real trigger path is used.
3. Cancel rides and check ledgers — cancellation adjustments vs completed-ride ledger behavior.
4. External payments and check ledgers — use mock-payment-server (or in-house mock) so payment/order flows update finance as in dev-shaped tests.

Reconciliation tie-in (later)
- DSSR_VS_SUBSCRIPTION: after (2) + scheduler trigger API (not raw scheduler_job INSERT only), allocator + producer running.
- DSR_VS_LEDGER: needs (1) and often (3) with data in the job date window.
- PG settlement recons: closer to (4) and settlement tables; larger scope.

Planned test-tool enhancements (single enhancement PR, slice as needed)
1. Flow simulation layer
   - Preset scenarios chained or one-shot; assertions on ledger rows / job status / recon summary where relevant.
   - Reuse context-api and dashboard flows; avoid duplicate SQL seeding where APIs exist.

2. API test section (mini Postman)
   - Method, URL, headers, JSON body against local base URLs (from dhall / same ports as nix stack).
   - Default auth: JUSPAY_ADMIN token (e.g. from atlas_bpp_dashboard.registration_token or documented env), with clear error bodies (4xx/5xx).
   - Optional: copy as curl; bind merchant/city from current dashboard context.

3. DB query runner
   - Read-only by default against local Postgres; schemas atlas_driver_offer_bpp (and rider as needed).
   - Saved snippets for common checks (finance_ledger_entry, subscription_purchase, scheduler_job, finance_reconciliation_summary).
   - Writes only behind an explicit toggle if ever added.

4. Integration
   - After a scenario run, suggest queries or pre-fill API panel from last context (merchant, city, time window).

Environment / ops note
- Processes and ports: see nix services (e.g. Backend/nix/services/nammayatri.nix) and dhall-configs/dev/*.dhall; reconciliation jobs need driver-offer-allocator + producer for Redis-based scheduler, and finance trigger via management API rather than DB-only inserts where possible.
- RECON_TRIGGER_API_BASE_URL / token env vars for context-api should align with provider-dashboard so trigger-recon uses the real createJobIn path.
