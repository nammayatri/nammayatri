02 · The Problem
Problem

The work around the work.

AI writes a feature in an afternoon — then we spend three days getting it to run. The dev loop didn't get faster, it got louder. Six structural bottlenecks, every sprint, every dev:

Shared QA bottlenecks
~3 PRs/week queued behind broken envs
One bad seed on master, ten devs blocked. PR queues form behind environments, not reviewers.
"QA is down again — can someone re-seed? My PR has been sitting for 6 hours."
today: shared QA→ Dev Box per dev
Flaky local stack
~90 min/day on env setup
Pg + Redis + Kafka + 4 services + 2 RN apps. Ports clash, containers die, RAM tops out.
"Why is 8013 already taken? I haven't started anything yet…"
today: , run-mobility-stack-dev→ pre-warmed Dev Box
No perf trendline
regressions caught in prod, not PR
p95 creeps up over 10 commits. No one notices until Grafana lights up on Monday morning.
"Latency on /rideSearch has doubled this month — when did it start?"
today: Grafana, post-hoc→ perf gate per commit
Config drift
~1 hotfix/week from local ≠ prod
Local seeds differ from master. Tests pass green here, red on staging, mystery in production.
"Works on my machine" — structural, not lazy.
today: hand-patched seeds→ one-click sync from master
Scattered launchers
6+ terminals open per dev
CC in one tab, driver RN in another, allocator in a third, mocks in a fourth. No single status, no single log.
"Which terminal printed that stack trace? Where did rider-rn even crash?"
today: tmux + memory→ Workflows tab
Opaque mocks
~½ day lost per new integration
Idfy / Juspay / OSRM — wire shapes live in tribal memory. Upstream changes break us silently.
"What's the response shape for verify_with_source? Who integrated it last?"
today: grep + slack archaeology→ captured + schema'd
Where we're going · Dev Box sequence (local laptop ↔ remote Nix Dev Box on tailnet)
Local laptop runs run-local-test-dashboard + the one backend / emulator / CC you're editing. The Nix Dev Box on tailnet hosts test-context-api, Caddy reverse proxy, and run-mobility-stack-dev — ports resolved by ports.nix. Dashboard pulls the service map via GET /api/test-ctx/services; client apps paste the base URL into Developer Mode and HTTPS-route through Caddy by URL prefix.

↳ AI removes the writing constraint. The bottlenecks left are infrastructure, signal, orchestration. The next 14 slides are about those three.

08 · Dev Boxes · VISION
Vision · Phase 2

Your laptop is for UI. The backend lives in the cloud.

A Dev Box is a per-developer container with the full backend stack pre-warmed. The dashboard + the one backend service you're actively editing run on the laptop; everything else lives in the box. Destroyed nightly, re-created in < 60s.

On Your Laptop
  • Test Tool Dashboard (React · :5173)
  • One backend service you're editing — e.g. rider-app w/ ghcid hot reload
  • RN apps in dev mode (rider · driver) attached to local metro
  • Editor / IDE
~3 GB RAM · zero local Postgres / Redis / Kafka
In Your Dev Box (remote)
  • Postgres + Redis + Kafka pre-warmed with synced config
  • All other backend services: driver-offer, allocator, dashboards, providers
  • Mock registry · idfy · juspay · osrm · sms · fcm
  • Reverse proxy with a single public endpoint (next slide)
spec'd per merchant · pooled · ~$0.40 / dev / day
Why now
AI lets a single dev touch services across three repos in one sitting. Their laptop can't run all of those at once. Dev Boxes are the only way to give every engineer a production-shape stack without each of them owning a Mac Pro.
09 · Dev Box Networking · VISION
Vision · Phase 2

Random ports inside. One stable URL outside.

The pattern
Every backend service inside the Dev Box starts on a random available port — no collisions, ever. A reverse proxy fronts them all on a single endpoint, routing by service-name prefix. Dashboard auto-discovers ports via a registry endpoint.

13 · Roadmap
Phased rollout

Each phase ships value on its own.

No "big bang." Every phase makes the dashboard immediately more useful while keeping the same UI, the same launch path, and the same single product story for the team.

Now
Collections runner · Coverage Overview · Remote Stack · Tools panel — all shipped.
Phase 1
Coverage as PR signal · perf snapshot per release commit · trend views · regression gating.
Phase 2
Dev Box MVP · random ports · reverse proxy with /svc/<name> routing · auto-discovery.
Phase 3
Workflows tab · CC/RN/allocator launchers · cross-app log piping with req-id correlation.
Phase 4
Mock schema inference · contract diffs · replay-as-baseline · upstream change alerts.
Later
Release gating on perf + coverage · Dev Box pools · shareable repros via permalink.
Owners
Dev-Experience squad · 2 backend · 1 frontend · part-time DevOps for Dev Box images.
Cost
Small. Dashboard, orchestrator, mock registry, config sync — already shipped. Remaining pieces are isolated services.
Risk
Mostly Dev Box ops cost & security model. De-risked by per-dev quotas and nightly destroy.
14 · Impact
Outcomes

What the team gets, week one.

For developers
  • Onboard in hours, not days. One UI explains the entire stack.
  • No more laptop fights. Heavy services live in your Dev Box.
  • Repro any bug with a link. Same URL pattern across all devs.
  • See perf + coverage in your PR. Before review, not after release.
  • AI velocity, finally cashed in. Code in minutes, run in seconds.
For the org
  • Release gates on real signals. Perf, coverage, integration pass-rate.
  • Config-driven determinism. Local = synced master = staging.
  • Caught at the mock layer. Upstream changes don't reach prod blind.
  • One UI to teach. One product to invest in. One owner.
  • Linear cost. Dev Boxes scale with the team, not with laptop generations.
The thesis in one line
Models keep getting smarter. Pipes around the model are how we turn that into shipped product. The Full Stack Dev Box is those pipes.
